## Recommended execution order

**001 → 002 → 003**, which is simply severity order.

- **001 first.** The only finding with a demonstrable structural defect: `width: "auto"` + `marginRight` are animated on a widget mounted in the **root layout** (`app/layout.tsx:238`), so the forced reflow is reachable from every route. It is also self-contained, and it removes one of the seven `framer-motion` imports that 003 would otherwise have to migrate — shrinking 003 from 7 files to 6.
- **002 next.** Delete-only, near-zero risk, and it is what users will *feel* most: Cmd+K is the highest-frequency interaction in the app.
- **003 last.** Pure hygiene with no user-visible effect. Its checklist is simplest once 001 has already removed one file from it.

## Dependencies between plans

- **003 depends on 001.** Plan 001 rewrites `components/whatsapp-widget.tsx` to pure CSS, removing its motion import entirely. If 001 runs first, 003 must **skip** that file — its checklist says to verify with grep, not assume. If 003 runs first instead, 001 will simply remove a `motion/react` import rather than a `framer-motion` one. Either order works; **001-first is less total work.**
- **002 is independent** of both and can be done at any time.

## Three corrections to the original audit

The audit table originally presented contained claims that **did not survive verification against the code**. All three were re-checked at commit 6699c06 before these plans were written. They are recorded here so nobody executes against a false premise.

### 1. The command palette's exit animation is dead code — VERIFIED

`components/command-palette.tsx` renders `{open && (...)}` (line 280), so the subtree **unmounts** on close. `data-open` is written in JSX only ever as the literal `"true"` (lines 283, 289) and is never bound to state. The `[data-open="false"]` rules at lines 160 and 174 **can never match**.

**Close is therefore already instant.** Only the 180ms *entrance* actually executes — and that is what plan 002 removes. The dead `-out` keyframes (which carry an `ease-in`, always a defect on UI) are deleted too, so the bad easing cannot be copy-pasted somewhere it *would* run.

Severity was reduced from HIGH to **MEDIUM** accordingly: the defect is real but half the size originally reported.

### 2. There is NO duplicate motion library, and no ~45 KB to save — VERIFIED

`motion@12.42.2` **declares `framer-motion@^12.42.2` as its own dependency** and is a thin re-export wrapper. A recursive scan of `node_modules` finds **exactly one** `framer-motion` directory (hoisted, `12.42.2`) and no nested copies. Both import styles resolve to the **same module instance**; npm already deduped them.

The audit's "~45 KB gz shipped twice" claim was **wrong and is retracted.** Plan 003 was downgraded from HIGH to **LOW** and reframed as consistency + dependency hygiene. Its expected bundle delta is **≈ 0**, its acceptance criterion is "within ±2 KB", and it explicitly instructs the executor **not to report a performance win.**

### 3. The WhatsApp widget finding stands — VERIFIED, and it is worse than first thought

`components/whatsapp-widget.tsx:54` animates `width` to `"auto"` **and** `marginRight` — two layout properties. Framer Motion cannot interpolate to `auto`, so each hover forces a **synchronous layout read**.

Newly verified: the widget is mounted in the **root layout** (`app/layout.tsx:15` and `:238`), so this is reachable from every route in the app — not just the landing page. This is why it is now **001**.

**No millisecond or kilobyte figure is claimed for it.** The forced reflow is a structural fact readable from the code; its actual cost **must be measured**, and plan 001 contains the before/after DevTools Performance protocol to do so.

## Measurement discipline

Two of the three original findings were overstated, both in the direction of claiming performance wins that the code did not support. Every plan here therefore states its performance claims as **things to measure, not things to assert**:

- 001 requires before/after DevTools Performance recordings; acceptance is *zero Layout blocks and zero forced reflows*, observed — not estimated.
- 003 requires before/after `next build` figures; acceptance is *±2 KB*, and a large improvement is to be treated as an **anomaly to investigate**, not a success.
- 002 makes no performance claim at all. It is a latency and dead-code fix.

## Design constraints all plans honour

From the project's North Star and `DESIGN.md`:

- **Instant high-frequency interactions.** 100+/day elements (Cmd+K) get no animation at all.
- **Transform / opacity / clip-path only.** Never `width`, `height`, `margin`, `padding`, `top`, `left`.
- **Reduced-motion compliance.** Movement dropped; comprehension-aiding state changes kept. *(Note: 001 and 002 each* add *reduced-motion handling where there was none.)*
- **Focus and accessibility preserved.** 002 in particular fences off all focus handling as out of scope — it is the most likely thing to break in a command palette.
- **No bounce, no springy overshoot, no gimmicks.** Press feedback in the 0.95-0.98 band; entrances start at 0.96, not 0.7.
- **OKLCH tokens / `color-mix()` only.** No raw `rgba()`.

## Not yet planned

Eight further findings from the audit remain open and unplanned:

- Press-feedback scales inconsistent across three components — 0.88 (`floating-dock.tsx:281`), 0.92 (`whatsapp-widget.tsx:23`), 0.93 (`dock-two.tsx:31`) — all outside the 0.95-0.98 band. *(001 fixes the widget's; the other two remain.)*
- `initial={{ scale: 0 }}` at `hero-product-reveal.tsx:132`, plus `initial={{ width: "0%" }}` at `:113`.
- Ungated hover motion on the nav dock (`dock-two.tsx:30`, `scale: 1.12`) — fires on touch tap.
- The framer-motion layer largely ignores `prefers-reduced-motion`: of ~13 motion components, only `dock-two.tsx:76` calls `useReducedMotion()`.
- A surviving bounce: `pop-in` in `globals.css:598-601` overshoots to `scale(1.06)` and is applied to `.pill-active`, despite a prior session shipping "Remove bounce easings".
- ~40 hand-typed `cubic-bezier()` literals where tokens already exist — and `--ease-out-expo` / `--ease-out-spring` (`globals.css:1879-1880`) are defined as the **identical value**.
- `components/smooth-scroll.tsx` is **never imported** (dead file) and is the only consumer of `@studio-freight/lenis`, which is still a declared dependency.
- `design-system/studyledger/MASTER.md` (lines 76, 93, 106) prescribes `transition: all 200ms ease` — and `CLAUDE.md` instructs applying MASTER.md to every component built, so the doc actively teaches a violation into all future code.
